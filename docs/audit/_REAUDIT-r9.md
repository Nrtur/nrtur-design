- **Part A:** `dealFieldsP` overrides the `stageName` field's options with the active pipeline's stage names. Verified live: on Customer Onboarding the Stage filter now offers Kickoff / In setup / Training / Live / Won.
- **Part B:** `dealViews`/`activeViewKey` are now per-pipeline maps keyed by `activeId`. Every pipeline (incl. new/custom-object boards) starts from the same full default set (`DEAL_VIEWS_DEFAULT` = All deals, My deals, Closing this month, Rotting deals, Big deals, Rotting big deals) but keeps its OWN editable copy — edits/saves in one pipeline never affect another. The view/filter resets on pipeline switch. Verified: compiles + boots, switching re-scopes the board + Stage filter, no runtime errors.

## R9-11 (user-reported) — the New Contact form and the Edit Contact drawer had drifted apart
`FIXED 2026-07-06` · **Medium-High** · Modeling defect (three parallel form definitions). The live create form comes from the **propSeed registry** (arCreateFields prefers it), `arDefaultFields` is a duplicate fallback, and the edit drawer/About panel is a third hand-written list (`cPrimary`/`cSecondary`) — they drifted:
- **Status was a fake control with the wrong vocabulary:** create offered the LEAD lifecycle (`New/Contacted/Nurturing/Qualified/Unqualified`) and `arBuildRecord` then discarded the choice, hardcoding `cstatus:'Prospect'`. Edit used the real `CONTACT_STATUSES` (`Prospect/Customer/Lost`). → Both now use `CONTACT_STATUSES`, default `Prospect`, and the picked value persists (validated).
- **Mailing address didn't round-trip:** create = free-text Long text; edit = structured street/city/state/zip/country grid whose `{...value}` spread corrupted a string into char-keyed objects. → New `Address` type: `arFieldKind`→`'address'`, ARField renders the identical grid, both field defs updated; DrawerField's grid now tolerates legacy strings.
- **Job title typeahead only worked in edit:** the SuggestInput wiring sat on the dead `arDefaultFields` fallback; the live registry field had no `suggest`. → registry `c_title` gets `suggest:COMMON_JOB_TITLES`.
- **Source options differed (6 vs 11):** single-sourced into `CONTACT_SOURCES`, used by create seed, fallback, and edit drawer.
- **Field order flips** (email/phone vs title/company blocks; address position): registry groups reordered so the create sequence mirrors the edit drawer: First/Last/Email/Phone/Job title/Salutation → Company/Owner/Status/Source/Tags → LinkedIn/Address.
Verified live (unit-tested the exact registry path): create order matches, Status=`['Prospect','Customer','Lost']` default `Prospect`, picked `Customer` persists to `cstatus` (junk falls back to `Prospect`), Source=11 shared options, title suggest present, address builds as a structured object. Deliberate non-goal: edit-only progressive fields (Seniority, Job function, Additional emails/phones, LinkedIn, Last activity/contacted) stay edit-only — create remains a quick-capture subset.

## HIGH — a shipped feature does not actually work

### (R9-1) Blueprint transition guard AND the Approval gate are bypassed on the kanban board, the list-view inline stage dropdown, and bulk stage-move
`FIXED 2026-07-06` · **High** · Modeling defect (guard on one of two write paths) — shared `guardStageMove(deal,toStage)` added in PipelinePage, called in `moveDeal`/`lvStage`/`onBulkStage`; scoped to primary-placement moves so custom pipelines aren't false-blocked. Verified: list-view Prospecting→Negotiation now blocked with the Blueprint toast, deal unchanged.
- Both gates live ONLY in the deal-detail `commitStage` (index.html:9168 blueprint, :9169-9173 approval).
- The board/list/bulk commit path `applyMove` (:8721) and its callers `moveDeal` (:8740), `lvStage` (:8808), `onBulkStage` (:8811) call `gateCheck` (:8739) — which checks **required fields only** (`dealMissingFor`) — never `blueprintAllows` / `needsApproval`.
- CONCRETE FAILURE: seed approval rule "deals over $50k need sign-off to close" (:~12217) and `DEAL_BLUEPRINT` (:7981, prospecting→only qualified/lost). Drag a $60k deal from Prospecting straight to Won on the board → it closes immediately, no allowed-transition check, no sign-off. The detail page blocks the identical move. Since drag-drop is the primary way deals move, both P6 subsystems are effectively inert.
- FIX: factor a shared `guardStageMove(deal,toStageKey,toStageName)` (blueprint + approval) and call it at the top of `moveDeal`/`lvStage`/`onBulkStage` (or inside `applyMove`) before any `setDeals`, enqueueing the approval request exactly as `commitStage` does.

### (R9-2) Rich text & Attachment field types are wired for DISPLAY but not for EDITING — creating/editing them mangles the value
`FIXED 2026-07-06` · **High** · Broken type→control pipeline — added `case 'Rich text':return 'richtext'` / `case 'Attachment':return 'file'` to `coCtrlType`, and `richtext`/`file` branches to `DrawerField` (render `RichTextField`/`CoFileField`). Verified live: `coCtrlType('Rich text')==='richtext'`, `coCtrlType('Attachment')==='file'`; components resolve; edit-draft/save preserve the value.
- `coCtrlType` (:22252) has no case for `'Rich text'`/`'Attachment'` → both fall to `default:'text'`. `CustomRecordDrawer` (create AND edit, :22507) renders every field through `coCtrlType`→`DrawerField`, so both new types become a plain single-line `<input>`. The real controls (`RichTextField` :2360, `CoFileField` :2336) are only reachable via `ARField`, which the custom-record drawer does not use.
- Display IS wired: `coCellRender` (:22518-22519) + `CoFieldDisplay` (:22528-22529) expect Attachment = `[{name,size}]` and Rich text = HTML.
- CONCRETE FAILURE: open the seeded "Meridian website rebuild" project → Edit. The Attachment field `f_docs` (an array) is fed to a text input → renders `"[object Object],[object Object]"`; any keystroke + Save overwrites the array with that string; the detail page then shows a chip literally reading `[object Object]`. Creating an Attachment offers no file picker at all — you get a bare string. Rich text shows raw `<b>…` markup in a text box.
- FIX: add `case 'Rich text':return 'richtext'; case 'Attachment':return 'file';` to `coCtrlType` AND matching `'richtext'`/`'file'` branches to `DrawerField` (render `RichTextField`/`CoFileField`), or route the custom-record drawer through `ARField`.

## MEDIUM

### (R9-3) Validation rules run ONLY on record create — edits and stage moves bypass them
`FIXED 2026-07-06` · **Medium** · `validateRecord` (:17778) invoked only in `doCreate` (:2491) + sheet create (:2540). The generic edit-drawer `save` (:~7143) and deal stage moves never call it. So "Sales-Ready leads need an email" / "Won deals need an amount" are enforced on a brand-new record but not when you edit an existing lead to Sales-Ready or close a $0 deal to Won. FIX: call `validateRecord(object,{...record,...patch})` in the edit-drawer save and in `commitStage`/`applyMove`.

### (R9-4) The board's `applyMove` still classifies Won/Lost with a fragile `/won|live/` name-regex (inconsistent with the fixed `dealTerminalKind`)
`FIXED 2026-07-06` · **Medium** · :8724 `_kind=/lost/.test(_nm)?'lost':/won|live/.test(_nm)?'won':null` drives which won/lost automations fire, while the same function's outcome uses the stable key-first `dealTerminalKind` (:4493). A custom stage named "Live" / "Delivery" / "Alive" fires "Deal won" automations though the deal isn't terminal — internally inconsistent. FIX: `_kind = dealTerminalKind(toStage)==='won'?'won':dealTerminalKind(toStage)==='lost'?'lost':null`.

### (R9-5) Automations match by trigger-NAME only, with no entity scoping — cross-entity misfire
`FIXED 2026-07-06` · **Medium** · `AUTO_INIT` entries carry no `object`; both buses filter on `events.indexOf(a.trigger)>=0` (:17866, :17951). `fireRecordUpdate` emits the bare `'Tag added'`/`'Tag removed'` for contact, lead AND company (:17881-17895). A Contact automation on `'Tag added'` therefore also fires when a tag is added to a lead/company, running contact-shaped steps on the wrong entity. FIX: stamp each automation with `object` and add `&&(!a.object||a.object===ent)` to both bus filters.

### (R9-6) Specific-tag triggers are dead
`FIXED 2026-07-06` · **Medium** · seed automation id:6 "Cold Lead Nurture" has trigger `'Tag added: cold-lead'` but the bus only ever emits bare `'Tag added'` — exact `indexOf` never matches. Any builder "Tag added: X" trigger never fires. FIX: normalize — treat `Tag added: <tag>` as event `'Tag added'` + a condition on the changed tag (the diff already knows which tag).

### (R9-7) Stage-specific triggers keyed by display name break on rename / custom pipelines
`FIXED 2026-07-06` · **Medium** · `runDealAutomations` emits `'Deal moved to '+toName` (:17962); seeds hardcode `'Deal moved to Proposal'`/`'…Won'`. Rename the Proposal stage → the automation silently stops (generic "stage changed" + won/lost still fire). FIX: match stage-specific triggers by `stageKey`, not display name.

### (R9-8) Per-automation Run-log modal shows fabricated rows, ignoring real runs
`FIXED 2026-07-06` · **Medium** · `logsForAuto` (:17974) reads the static `AUTO_LOGS` seed and, on no match, synthesizes fake rows from `LOG_SAMPLE_NAMES`; the live `autoLogs` state is ignored (the main Logs panel at :18203 correctly merges it). A real run is invisible in that automation's own modal, and fake people appear. Violates the project's no-fake-data rule. FIX: `const real=[...(autoLogs||[]),...AUTO_LOGS].filter(l=>l.auto===auto.name)` and synthesize only when both empty.

### (R9-9) Rich text rendered via unsanitized `dangerouslySetInnerHTML`
`FIXED 2026-07-06` · **Medium** (Low in prototype context) · :22528 / :2374 inject stored HTML with no sanitizer (repo-wide: no `sanitize`/`DOMPurify`/`escapeHtml`). Combined with R9-2 forcing raw-HTML authoring, `<img src=x onerror=…>` in a Rich text field executes on any teammate's view. FIX: a shared `coSanitizeHtml()` (tag/attribute whitelist) used by `RichTextField.sync()` and `CoFieldDisplay`.
