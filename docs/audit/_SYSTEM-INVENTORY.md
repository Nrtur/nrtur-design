## Contact

**What it is for:** A real person record in the CRM — the people you actually work and sell to. Holds identity (name/email/phone/title), a customer-lifecycle status (Prospect/Customer/Lost via cstatus), an owner, tags, a lead-score, ad/source attribution, DNC/suppression flags, and address. It is the anchor for the record-detail page's activity timeline, tasks, deals, and invoices. Contacts are created directly, via CSV/quick-add/record-sheet, or produced when an ad-lead is captured (contactId on the ad-lead points back here).

**Smells flagged for the gap analysis:**
- TWO status fields on Contact: `cstatus` (real enum Prospect/Customer/Lost) and `status` (Active/Hot/Warm/New/Lead/Qualified). `status` is vestigial — never rendered or edited on any Contact screen, hardcoded to 'Active' on create (@2334). Dead/confusing field.
- FUSED LEAD/DEAL CONCEPT inside Contact: the vestigial `status` holds 'Lead' (ids 120/7/9/10) and 'Qualified' (id 8) — Lead-lifecycle values living on a Contact record. 'Qualified' is the classic fused value appearing across lead + deal + here. Contacts should never carry a lead lifecycle.
- Lead-qualification artifact embedded on a Contact: id:1 has a `qual` object with status:'qualified', a form, an event and a rep — qualification belongs to the Lead lifecycle before conversion, not persisted on the resulting Contact. QualBadge(qual.status) is only wired to the Lead detail page (@8909), never shown on the Contact page, so it's dead data here.
- Company link is name-string only (contact.co), no id. Renaming a Company or a typo silently orphans the contact; ContactCompanyField/onCreateNew can spawn a duplicate company by name. Wrong-owner risk: the join is by string equality.
- contactsForCompany@8086 (and dealsForCompany) read the module-level constant CONTACTS_DATA / allDeals(), NOT live React state — company detail 'contacts' and counts won't reflect contacts created/edited/archived at runtime. Stale-data defect.
- 'Open company' from the contact detail (companylookup go:g=>g('company-detail')) navigates with NO id — it opens a default company, not this contact's company. Broken cross-navigation.
- Deals panel on the Contact detail (@7303-7304) is fully HARDCODED (count 'Deals · 2' + two fake deals Q3 Engagement/Annual Retainer with stages Proposal/Qualified) — identical for every contact and disconnected from the real name-string deal derivation the ActivityTimeline uses (@6767). The two disagree; the visible panel is fake.
- `deal:'$24k'` on the Contact is a display string, not a real Deal reference — implies a 1:1 contact-to-money value on the Contact when deals should live as separate Deal records (1:many). Money/pipeline modeled on the wrong entity.
- Duplicate contacts by design: ids 1 & 101 (both 'Sarah Chen'/Meridian) and 3 & 102 (both 'Maria Lopez') — name-string joins to Deals/Tasks/Invoices are ambiguous across duplicates (which Sarah Chen owns the deal?).
- Contact stage/pipeline: there is NO pipeline on Contact (correct) — but the vestigial `status` with 'New'/'Qualified' mimics a pipeline-ish progression and invites confusion with the Deal pipeline and Lead lifecycle.
- Owner is stored as avatar-initials string (e.g. 'AM'), not a user id — fragile join to TEAM_MEMBERS by initials.

**Smells flagged for the gap analysis:**
- DUAL-MODE (name-string vs id) RELATIONSHIP STORAGE — the classic fragile link. Seed records store target NAMES (r_company:'Meridian Agency', r_contacts:['Sarah Chen'] @20460) while CoLinkField stores target IDs for any record created/edited in the UI (toId @20576-20578). coResolveLink papers over this by resolving id OR name (@20429). Consequence: seed/legacy links are name-keyed → break silently on rename and collide on duplicate names; new links are id-keyed → rename-safe. Same field, two incompatible key schemes. Comment @20414-20415 admits it ('Backward-compatible — legacy name-keyed values still resolve by name').
- PIPELINE OWNED BY A CUSTOM FIELD, NOT BY A DEAL. A CustomObject can be driven as a full pipeline (ObjectBoard @7575,7582; NewPipelineModal @7664) by overloading an arbitrary Single-select field (e.g. Projects.f_stage) as stages. Deleting/renaming a 'stage' actually edits the field's options and rewrites record values by string match (@7628,7630). This is the pipeline living somewhere other than a Deal — acceptable as an operational board, but it means 'stage' semantics are duplicated across Deals, Leads, Contacts, Companies AND every custom Single-select, with no shared stage model.
- 'Stage'/'status' concept fusion risk. Nothing stops a user from naming a custom Single-select field 'Stage' with options like Qualified/Won — replicating deal-pipeline vocabulary on an unrelated object, and 'Qualified' can collide with Lead lifecycle wording. No guardrail on option naming.
- LINKS ARE ONE-DIRECTIONAL / NO BACK-REFERENCES. A custom record stores its outbound links, but the target (Company/Contact/Lead/Deal/other custom object) stores nothing pointing back. CoLinksPanel (@20645-20656) only renders the current record's own rels; the linked Contact's detail page will NOT show 'linked from Project X'. Relationship integrity is therefore asymmetric and unenforced (deleting the target leaves a dangling name/id that coResolveLink renders as plain text with nav:null @20430).
- CUSTOM RECORD CAN LINK TO A LEAD — boundary smell. 'lead' is an allowed relationship target (@20487, coTargetOptions filters out converted leads @20421). A user-defined object attaching to a top-of-funnel Lead blurs the lead-vs-real-record boundary; and because links are one-directional + name-fragile, a Lead that later converts (its name/id changes context) can orphan the custom record's link.
- OWNER KEYED BY AVATAR INITIALS, NOT USER ID. record.owner stores TEAM_MEMBERS avatar (e.g. 'AM') and coMember matches on avatar (@20433). Two members with identical initials collide; ownerColor is denormalized/recomputed (@20598) rather than referenced.
- DELETE-OBJECT IS UNGUARDED AGAINST INBOUND LINKS. delObj (@20824) permanently removes an object and its records with only a name-confirm; any OTHER custom object that had a relationship targeting it (target=that object's id) is left with dangling links that resolve to bare strings — no cascade or referential check.
- LINKED-RECORD ACTIVITY NOT SURFACED. Unlike deals (which pull linked contact/company activity into the timeline @6780-6784), actCustomOwn (@20437) only shows the record's own created event + rec.acts. A custom record linked to a Company/Contact shows none of that entity's activity — a missing-piece vs the standard-object timelines.
- PERMISSIONS ARE A PROXY, NOT REAL. effCanCustom just aliases effCanObject('Contacts',act) (@20387) and list-scoping uses effScopeRows('Contacts',...) (@20692,1650). Every custom object inherits the Contacts role capability regardless of its own sensitivity; the only per-object control is the binary visibility:'admins' lock (@20474).

**Smells flagged for the gap analysis:**
- FUSED CONCEPT / mislabeled object: 'Ad Leads' do NOT create Leads. ingestAdLead@21192 writes to crm.contacts (cstatus:'Prospect'), and the _adLeads@8522 array is a parallel attribution log keyed by contactId — so the top-of-funnel entity that SHOULD be a Lead is a Contact, and 'lead' in the module name is a misnomer.
- INCONSISTENT capture target: Forms create EITHER a Contact (form1 recordType:'contact'@16588) OR a Lead (form4 recordType:'lead'@16597) for near-identical inbound intent; forms 2/3/5 have no recordType at all. Booking@18358 always creates a Lead; Ad ingest always creates a Contact. Four different modules disagree on whether an inbound stranger is a Lead or a Contact.
- PIPELINE OWNER on the wrong entity: Ad-lead forms store pipeline:'Sales Pipeline'+stage:'New'@8481 (a DEAL pipeline stage) on the lead-capture form, and 'New' collides with LEAD_STATUSES 'New' — a deal-stage name bolted onto a lead-capture object. The stage is also dead config: ingest never creates a deal.
- FUSED 'Qualified' again: Forms 1&4 set dealStage:'Qualified'@16588/16597 and booking stageKey:'qualified'@18369 — the classic lead-lifecycle vs deal-pipeline 'Qualified' overload, now reached straight from a form/booking that skips the lead stage entirely.
- DEAL created without lead lineage: Form createDeal@16588 and Booking createDeal@18370 spawn a Deal referencing the person only by name-string (company:nm, primaryContact:nm) — the Deal is not tied by id to the Lead/Contact created in the same transaction, so the two records share no key.
- MONEY not attached to Deal: booking 'pay-at-booking' invoice sets dealId:null@18382 even when a Deal was just created@18370; payment links@19117 have no owner at all. Revenue can't reconcile to the opportunity it belongs to.
- INBOX is a pure display facade: email.contact@8988 and email.linkedDeal are nested literal objects, SMS convo.contact@9696 is a name+phone string — none carry an id, none are looked up in crm.contacts/deals. The 'auto-log to the right contact and deal' promise (9864 copy) is not backed by any id link; only runtime nav.contactId@9828 bridges when you arrive FROM a contact.
- SEQUENCE enrollment is not a relationship: SEQ_DATA.enrolled@11563 is a static integer; enrollment shows only as a synthesized timeline activity@6688/6708 gated by a boolean (c.adEnrolled). No table maps sequence→contact ids, so 'who is in this sequence' and 'unenroll X' are unrepresentable.
- PAYMENTS→Contact/Company are name-string joins@19336/6773: invoice.contact/company resolve via find(c=>c.name===...) — a renamed or duplicate-named contact silently detaches its invoices from the timeline. Only invoice→Deal (dealId) is a clean id link.
- MEETING→Deal is display-only: meeting.dealName/dealValue@18564 are strings with no dealId; the meeting knows its contact/lead by id but the deal it advances is untethered text.
- TWO disjoint FORM systems: FORMS_DATA@16585 (site forms, numeric ids) vs AD_LEAD_FORMS_SEED@8480 (ad forms, lf_* ids) never share a schema or namespace; funnelDefaultFormId@19720 also hard-defaults every funnel form block to FORMS_DATA[0], so newly built funnels silently point at form #1.
